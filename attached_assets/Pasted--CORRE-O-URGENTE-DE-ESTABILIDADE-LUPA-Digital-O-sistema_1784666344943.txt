==================================================
PROBLEMA 1 — INTERPRETAÇÃO SIMULADA
==================================================

A interface está exibindo:

“Interpretação simulada
IA temporariamente indisponível.
Usando interpretação por palavras-chave.”

Isso não deve acontecer quando o provedor Llama estiver disponível.

Verifique:

- qual provedor de IA está ativo;
- qual variável de ambiente seleciona o provedor;
- se a chave da API está configurada;
- se o modelo está correto;
- se o backend consegue chamar o provedor;
- se existe erro de autenticação;
- se existe erro de limite;
- se existe timeout;
- se a resposta está chegando;
- se o parser JSON ou o schema Zod estão rejeitando a resposta;
- se uma exceção está ativando o fallback de forma indevida.

Objetivo:

- usar IA real sempre que o provedor estiver disponível;
- ativar o fallback por palavras-chave apenas em indisponibilidade real;
- registrar internamente a causa exata do fallback;
- não mostrar ao usuário uma análise simulada como se fosse equivalente à análise por IA.

Na interface, quando a IA estiver indisponível, mostrar:

“Serviço de IA temporariamente indisponível.
Tente novamente em alguns instantes.”

Não gerar resultado simulado automaticamente, a menos que exista uma configuração explícita de demonstração.

==================================================
PROBLEMA 2 — CONEXÃO COM POSTGRESQL
==================================================

O erro atual é:

failed to connect to postgres:
failed to connect to `user=postgres database=postgres`
dial error: timeout: context deadline exceeded

O endereço mostrado é IPv6 e a conexão expira.

Investigue:

- DATABASE_URL;
- DIRECT_URL;
- URL do pooler do Supabase;
- porta utilizada;
- SSL;
- IPv4/IPv6;
- senha;
- host;
- formato da connection string;
- uso incorreto da conexão direta em ambiente que não suporta IPv6;
- cliente PostgreSQL utilizado;
- prioridade entre DATABASE_URL e DIRECT_URL.

Regras:

1. Para execução normal no Replit e em produção serverless, priorizar a URL do pooler do Supabase compatível com IPv4.
2. Usar DIRECT_URL apenas para operações que realmente precisem de conexão direta, como migrations, quando suportado.
3. Garantir `sslmode=require` ou configuração SSL equivalente.
4. Não expor usuário, senha ou connection string nos logs.
5. Implementar timeout de conexão controlado.
6. Se a conexão falhar, a aplicação não deve travar completamente quando a operação puder funcionar apenas com a API Supabase.
7. Criar um health check seguro para confirmar:
   - banco acessível;
   - latência;
   - tipo de conexão utilizada;
   - sem exibir secrets.

Não tente corrigir esse erro criando banco PostgreSQL local.
Não use sudo.
Não altere a senha automaticamente.

==================================================
PROBLEMA 3 — ERRO GENÉRICO NA INTERPRETAÇÃO
==================================================

A interface está exibindo:

“Não foi possível concluir a interpretação.
Não foi possível concluir a interpretação neste momento. Tente novamente.”

Essa mensagem esconde a etapa real que falhou.

Audite o fluxo completo:

entrada do documento
→ extração de texto
→ normalização
→ verificação de tamanho
→ montagem do prompt
→ chamada ao Llama
→ recebimento da resposta
→ extração de JSON
→ validação Zod
→ persistência
→ resposta ao frontend

Identifique exatamente onde a falha ocorre.

Mapeie erros para mensagens específicas:

- documento vazio:
  “Não foi possível localizar texto no documento.”

- PDF digitalizado:
  “O PDF não possui texto pesquisável. Utilize OCR ou outro arquivo.”

- documento grande:
  “O documento ultrapassa o limite da análise. Ele precisa ser processado em partes.”

- provedor indisponível:
  “O serviço de IA está temporariamente indisponível.”

- limite de uso:
  “O limite temporário de análises foi atingido. Aguarde e tente novamente.”

- timeout:
  “A análise demorou mais que o esperado. Tente novamente.”

- resposta inválida:
  “A IA retornou uma resposta incompleta. A análise não foi salva.”

- banco indisponível após análise:
  “A interpretação foi concluída, mas não foi possível salvá-la no histórico.”

Não expor ao usuário:
- stack trace;
- nome de organização;
- chave;
- URL privada;
- mensagem completa do provedor.

Usar status HTTP coerentes:
- 400 para entrada inválida;
- 413 para documento muito grande;
- 422 para resposta não validável;
- 429 para limite;
- 502 para resposta inválida do provedor;
- 503 para indisponibilidade;
- 500 apenas para falha interna inesperada.
